DIW: DELVING INSTANT WEB-SEARCH v1.4.o

The Delving Instant Web-search (DIW) is a javascript powered, stand-alone interface used for remotely querying a Delving Hub3 v1 API. The DIW has a configurable endpoint to which the queries are made and has many more configuration options to customize the outputted results to your own web environment.

NOTE: This version of the DIW is not compatible with legacy hubs.

jQuery

The DIW depends on jQuery to function. Currently the DIW is written to use jQuery version v3.1.0 which is included with the distribution.

IMPORTANT If your web environment also uses other libraries like prototype.js, be sure to include <script>jQuery.noConflict();</script> after loading the DIW javascript files.

Styling with css

The DIW only needs one stylesheet to function properly: the delving-instant.core.css. Using the included bootstrap.css is optional. For the rest, the styling is completely up to you.

Depending on what the DIW is doing, it will write its current state as a class name into the body tag. States are:

  • diw-init - The DIW is initialized, but no query has been performed;
  • diw-search-results - The DIW has performed an initial search request and is displaying the results;
  • diw-detail - The DIW is showing the detail view on a single search result;

You can use these body classes to gain more fine-grained control over the layout.

Deep-Zoom

The DIW is capable of showing DeepZoom (.dzi) image tiles. It uses the OpenZoomViewer.swf (a flash file) as the default viewer. If the user's browser is not capable of using flash, then there is a fallback to the Seadragon javascript library which is included in the delving-instant-dependencies.js. The flash version and the javascript version of deepzoom each have their pros and cons. It is up to you to pick the best match for your environment or you can choose not to use deepzoom at all and only show static images. More about deep-zoom here.

Setup

  1. Include the delving-instant.core.css in the head of your html document. You can also add the bootstrap.css for Twitter Bootstrap styling, and of course, your own custom stylsheet with which you can override any of the delving-instant.core.css or bootstrap.css styles. Always include your own custom stylesheet last in order for overrides to work.

  2. Include the delving-instant-dependencies.js, delving-instant-core.js and delving-instant.config.js (in that order) at the end of your document before the </body> tag.

  3. Set the parameters in the delving-instant.config.js to reflect your pages structure, naming conventions and endpoint Hub settings.

IMPORTANT: If you are not sure what to do with these parameters, please contact your Hub administrator.

A basic html (html5) page running the Delving Instant should might look something like this:

<!DOCTYPE HTML>
<html>
<head>
    <title>Delving Instant Website</title>
    <meta name="author" content="Delving B.V. http://www.delving.eu" />
    <meta http-equiv="Content-Type" content="text/html; charset=utf-8" />
    <link rel="stylesheet" type="text/css" media="all" href="css/delving-instant-core.css"/>
    <link rel="stylesheet" type="text/css" media="all" href="css/bootstrap.css"/>
    <link rel="stylesheet" type="text/css" media="all" href="css/delving-instant-custom.css" />
</head>
<body>
    <div id="diw-search"><!-- autogen search-form here --></div>
    <div id="diw-advanced-search"><!-- autogen search-form here --></div>
    <div id="diw-query-info"><!-- autogen query info here --></div>
    <div class="diw-pagination"><!-- autogen pagination here --></div>
        <div id="diw-results-container">
            <div id="diw-facets"><!-- autogen facets here --></div>
            <div id="diw-results"><!-- autogen results list or grid --></div>
        </div>
        <div id="diw-item-container">
            <div id="diw-item-media"><!-- autogen image/deepzoom here --></div>
            <div id="diw-item-data"><!-- autogen item metadata here --></div>
            <div id="diw-item-related"><!-- autogen related items here --></div>
        </div>
    <div class="diw-pagination bottom"><!-- autogen pagination here --></div>
    <script src="js/jquery.min.js"></script>
    <script src="js/delving-instant-dependencies.min.js"></script>
    <script src="js/delving-instant-core.min.js"></script>
    <script src="js/delving-instant-config.js"></script>
    <!-- Use this if using other libraries like prototype.js -->
    <script>jQuery.noConflict();</script>
</body>
</html>

All the html block level div elements you see above are placeholders for content generated by the DIW. The id's and classes used above are the default ones used by the DIW to know where to write different bits of information to. You can override these in the configuration for the DIW and associate your own block-level elements with DIW output. This is done in the layout configuration option.

Configuration options

An important note about configuration:

Some configuration options are in the form of string arrays and while others are object arrays

String array
collectionSpecs: [
    "spec_name_1",
    "spec_name_2"
],
Object array
itemFields: [
    {name:"field_name_1",label:"Label 1 text"},
    {name:"field_name_2",label:"Label 2 text"},
    {name:"field_name_3",label:"Label 3 text"}
],

What is important to note here is that the last item in the array is NOT followed by a comma. Most browsers (if the last item does have a comma after it) will still interpret the data properly and the DIW will function as it should. But Internet Explorer will choke over this last comma. So please make sure that your configuration arrays are properly formatted.

configHelp (boolean)

With this set to true you will see an area on the top-right side of the page containing the following information:

  • Link with URL to the actual search response
  • List of available facets
  • Link with URL to the item data
  • List of available field names
  • Possible field errors

Use this option to determine which fields are available before contacting a HUB administrator.

Note: Having configuration help turned on will slow down the response times of the application.

ATTENTION: If you have this set to true in your configuration make sure to disable it for production environments! 

configHelp: true|false

usePushState (boolean - default: true)

The DIW takes advantage of the history pushState which visibly changes the URL in the location bar to perform search actions, but the actual page is not reloaded. This may cause issues in older browsers or certain CMS environments. You can force the page to physically reload by setting this configuration to false

usePushState: true|false

endPoint (string) (mandatory)

Full base URL (incl. 'http://') for the API and NO trailing slash

endPoint: "http://playground.delving.org"

orgId (string) (mandatory)

The system name of the organization under which the dataprovider is registered in the endpoint Hub

orgId: "delving"

collectionSpecs (string array)

The 'spec' (system name) of the the specific collection(s) you want to limit the queries to. If excluded, all collections are queried.

collectionSpecs: [
    "spec-1",
    "spec-2"
],

rows (integer - default: 12)

Number of result items returned per page.

rows: 12

useSimpleSearch (boolean - default: true)

Toggle the view of the of the simple search box. In some cases you may wish to create your own.

useAdvancedSearch (boolean - default: false)

Setting this to true will activate the advanced search form. It is hidden by default, but a link will appear next to the basic search form where the visibility can be toggled.

useAdvancedSearch: true|false

advancedSearchFields (object array)

Here you can set the list of fields to be used for the advanced search. These fields will be used to generate a dropdown from which users can select. If you are not sure which fields are applicable then contact your Hub administrator.

advancedSearchFields : [
    {name: "", label: "Any"},
    {name: "dc_date_text", label: "Date"},
    {name: "dc_title_text", label: "Title"},
    {name: "dc_type_text", label: "Material"},
    {name: "dc_subject_text", label: "Subject"}
],

Note: field names are appended with "_text" to define an advanced search field.

label (object)

The label object contains a list of key/value pairs for labels/texts that are used in the interface.

label: {
    advancedSearch: "Advanced search",
    advancedSearchClose: "Close advanced search",
    collapse: "Less [-]",
    expand: "More [+]",
    viewInOriginalContext: "View in original context",
    goTo: "Go to: ",
    next: "Next",
    noResults: ":-( nothing found",
    previous: "Previous",
    numberOfResults: "Number of results: ",
    returnToSearchResults: "Return to search-results",
    search: "Search",
    searchResults: "Search-results",
    relatedItems: "Related items",
    sort: "sort",
    sortNumeric: "0-9",
    sortAlphabetic: "A-Z"
},

layout (object)

Sets the html block level elements where you would like the DIW to write its output to.

layout: {
    simpleSearchBlock: "#diw-search",
    advancedSearchBlock: "#diw-advanced-search",
    queryInfoBlock: "#diw-query-info",
    paginationBlock: ".diw-pagination",
    resultsBlock: "#diw-results",
    facetsBlock: "#diw-facets",
    itemMediaBlock: "#diw-item-media",
    itemDataBlock: "#diw-item-data",
    itemRelatedBlock: "#diw-item-related"
},

The simpleSearchBlock is where the basic search form is written to.

You can also create your own form to launch a DIW query (you may wish to be able to launch queries from other pages). All that is necessary is that the form's action parameter points to page containing the DIW files and that the text input's name attribute is set to "query" and the method to "GET".

<form method="get" action="your_diw_page">
<input type="text" name="query" required="true" />
<input type="submit" value="search"/>
</form>

Set the useSimpleSearch configuration to false if you want to create your own search form on the page.

The advancedSearchBlock is where the advanced search form is written to.

The queryInfoBlock is where the number of results and query crumbs are written to.

The paginationBlock is where the pagination will be written. A class is used instead of an id because you may want to have two pagination areas (at the top and bottom of the page).

The resultsBlock is where the search results are displayed.

The facetsBlock is where the facets are displayed.

The itemMediaBlock is where the item image (or zoom image) are shown.

The itemDataBlock is where the items metadata fields are displayed.

The itemRelatedBlock is where the related items are displayed.

resultsLayout (string - default: "list")

You can choose one of two ways to render the initial results: "grid" or a "list". The grid allows for a more visual experience but displays less metadata. The list option is visually less dynamic but will show more initial data (if available).

resultsLayout: "list"

language (string - default: "en")

Certain elements can be returned as internationalized fields (like the field names when viewing a single result item).
NOTE: This will only function if you do not configure the itemFields block where you can supply your own label names.

language: "nl"

itemId (string - default: "diw-id")

URL parameter for linking to a single item. Made configurable to resolve eventual conflicts with web environments where the "id" parameter is already in use.

itemId: "diw-id"

pageId (string - default: empty)

Name of the id parameter used to identify a page within an environment like Wordpress or Drupal. Set this inorder to maintain the page location on a hard refresh.

pageId: "page_id"

textExpander (boolean - default: false)

Turn text-expansion on or off (defaults to false). If on, then maxCharacterResults and maxCharctersItems should be set

textExpander: true|false

maxCharactersResults (integer - default: 500)

The maximum number of characters for fields within the initial results. If a field is longer, it will be collapsed after this many characters and a link to expand the text will be created.

maxCharactersResults: 250

maxCharactersItems (integer - default: 1200)

The maximum number of characters of a returned field for a single item. If a field is longer, it will be collapsed after this many characters and a link to expand the text will be created.

maxCharactersItems: 750

maxCharactersTitle (integer - default: 100)

For titles, the length is set separately because they are often in the form of html links. This fact makes them unusable with the text-expansion functionality. To still give you control over the maximum display length of this field, this parameter is supplied. The title's length will be limited to this many characters and then appended with ellipses.

maxCharactersTitle: 100

thumbnailField (string - default "delving_thumbnail")

The fieldname containing the thumbnail.

thumbnailField: "delving_thumbnail"

thumbnailWidth (integer - default: 144)

Width (in pixels) for the images returned in the initial search results grid or list. This value adds the width parameter to the image. It can be overwritten by custom css if so desired.

thumbnailWidth: 200

resizeThumbnails (boolean - default: false)

resizeThumbnails: true|false

Toggles the use of imgLiquid.js - https://github.com/karacas/imgLiquid - which will stretch and fill the thumbnail area in the results grid - this will not affect the list layout. This setting also overrides the thumbnailWidth setting.

useDeepZoom (boolean - default: false) - set for deprecation

If the data contains deep-zoom urls (.dzi) you can set this to true to activate the deep-zoom functionality.
Default: false
IMPORTANT: Merely setting this to true will not make the images zoomable. The metadata must contain functioning links to the deep-zoom tiles. The actual metadata field containing the deep-zoom uri can be configured here.

useDeepZoom: false

NOTE The use of flash deepzoom will be deprecated. OpenSeadragon is more secure and offers more features.

useFlashZoom (boolean - default: true) - set for deprecation

By setting this to true deep-zoom images will be loaded into the OpenZoomViewer.swf included in the DIW. If the user's browser is not capable of using Flash the DIW will revert to using the Seadragon javascript deep-zoom library also included in the DIW dependencies file. For best results in Internet Explorer set this to true

useFlashZoom: true|false

deepZoomFlashFile (string) - set for deprecation

The path to the OpenZoomViewer.swf file.

deepZoomFlashFile: "flash/OpenZoomViewer.swf"

deepZoomControlImages (string)

The path to the Seadragon control panel images. Defaults to: "images/seadragon/"

deepZoomControlImages: "path/to/deepzoomControlImages/"

defaultImg (string)

Location of default image (used when search result's thumbnail cannot be found). Defaults to "images/blank.png".

defaultImg: "path/to/img.png"

collapseFacets (boolean)

When set to 'true' the facets will only show the headers. When click will expand to show all facets under that header. Defaults to 'false'

collapseFacets: true

sortFacets (boolean)

Let the user sort facets alphabetically or numerically (by number of hits). Defaults to 'false'

sortFacets: true

Note: with configHelp set to true you will get a list of available fields for facets, results and item display. The availability of the field in the api does not guarantee that the field contains data.

displayFacets (object array)

List of facets to be displayed, and the label to display above the facet group. Facets are displayed in the order they are listed here.

displayFacets: [
    {"name":"dc_date_facet","label":"Date"},
    {"name":"dc_type_facet","label":"Object type"},
    {"name":"dc_subject_facet","label":"Subject"},
    {"name":"nave_color_facet","label":"Color"}
]

customFacets (string array)

Array of custom facets defined in the Hub index for a particular organization. Check with your Hub administrator which custom facet strings are available.

customFacets: [
    "custom_facet_one",
    "custom_facet_two",
    "custom_facet_three"
]

Note: The field name for a facet configuration is appended with "_facet"

Note: When adding custom facet fields be sure to also add the corresponding facet object to the displayFacets. So if adding a custom facet for dc_format, then your configuration should look something like this:

customFacets: [
    "dc_format_facet",
],
displayFacets: [
    {"name":"dc_date_facet","label":"Date"},
    {"name":"dc_subject_facet","label":"Subject"},
    {"name":"dc_format_facet","label":"Format"}
]

resultFields (object array)

These are the fields that will be displayed on the initial search results (if "list" is set for resultsLayout - "grid" only shows a title and no other fields). You can leave the label empty if you only want to display the field value (usually for titles). Labels and values will appear in the order that you list them here.

resultFields: [
    {name:"dc_title", label:""},
    {name:"dc_creator", label:"Creator"},
    {name:"dc_date", label:"Date"},
    {name:"dc_subject", label:"Subject"},
    {name:"dc_description", label:"Description"}
],

itemFields (object array)

List of field names to be displayed for the item view. If a field is not present in the returned meta-data it will not be displayed. Fields are displayed in the order they are listed.

Basic usage
itemFields: [
    {name:"field_name_1",label:"Label 1 text"},
    {name:"field_name_2",label:"Label 2 text"},
    {name:"field_name_3",label:"Label 3 text"},
    {name:"field_name_4",label:"Label 4 text"},
    {name:"field_name_5",label:"Label 5 text"},
    {name:"field_name_6",label:"Label 6 text"}
]
Advanced usage
itemFields: [
    {name:"field_name_1",label:"Label 1 text"},
    {name:"field_name_2",label:"Label 2 text", newQuery:true},
    {name:"field_name_3",label:"Label 3 text"},
    {name:"field_name_4",label:"Label 4 text", newQuery:true},
    {name:"field_name_5",label:"Label 5 text"},
    {name:"field_name_6",label:"Label 6 text"}
]

The newQuery parameter will turn this field into a hyperlink which, when clicked, will launch a new query for that value.

alternativeFieldLabelsPerSpec

If querying multiple collections, you may wish to use different labels for the same fields. For example: both collections A and B display the dc_creator field. However, collection A is bibliographic data and the label should be "Author", while collection B is photographic data and the label should be "Photographer. To accomplish this you will need to know the "spec" of the dataset. Once you know the dataset spec you can define the alternative labels. These alternative labels will be shown in both the initial search results defined in resultFields (if "list" is set for resultsLayout) and for the fields show in the item detail view defined in the itemFields.

alternativeFieldLabelsPerSpec: [
    {spec:"spec-a", fields:[
        {name:"dc_creator", label:"Author"},
        {name:"field_xyz, label:"abc"}

    ]},
    {spec:"spec-b", fields:[
        {name:"dc_creator", label:"Photographer"}
    ]}
]

alternativeFieldLabelsPerSpecTrigger

The field which distinguishes one collection from another. In most cases you will not have to set this and the default field is used. In some cases (merged collections) the default field will be the same for the collections involved. In this case you can supply an alternative field which will be used to distinguish between the collection. Note: this field must be present and unique in all collections. Contact your Hub administrator if you do not know what field to use.

alternativeFieldLabelsPerSpecTrigger: "field_name"

itemImageFields (object)

The fields related to the item image. This is mainly for the thumbnail, regular image, deepzoomImage, and navigation thumbs when an object contains multiple images. This is also where the link to the item in its original context is rendered. Notice the 'imageAlt'. Here you can supply an alternative field for the large image to display as a fall-back option if the 'image' field is not available. This is usefull if you have configured the DIW to query mulitple collections that are not identically mapped.

itemImageFields: {
    thumb:"europeana_object",
    image:"europeana_isShownBy",
    imageAlt:"europeana_object",
    deepZoomUrl:"delving_deepZoomUrl",
    externalLandingPage:"europeana_isShownAt"
}

showRelatedItems (boolean)

Setting this to true displays a list of items related to item currently being viewed.

showRelatedItems: false

relatedItemsCount (integer)

The number of related items returned.

relatedItemsCount: 5

Twitter Bootstrap

The DIW is 'bootstrapped'; meaning that many of the functional html sections contain Twitter Bootstrap classes. By including a boostrap.css file you will get all the Twitter Bootstrap goodness. Note: include the bootsrap.css file AFTER the diw-core.css file to overwrite some of the bootsrap styles (particulary the image carousel) which is also defined in the delving-instant-core.css file. You can create your own custom bootstrap.css file here: Twitter Bootstrap

When customizing the bootstrap.css the following components should be checked:

Scaffolding
  • None
Base CSS:
  • Forms
  • Buttons
Components
  • Button groups and dropdowns
  • Navs, tabs and pills
  • Breadcrumbs
  • Pagination
  • Thumbnails
  • Media component
JS Components
  • Carousel
Miscellaneous
  • Utilities
Responsive
  • None

None of the plugins need to be checked (The carousel plugin is already included in the delving-instant-dependencies.js)
